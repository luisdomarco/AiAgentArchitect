## 1. Handoff S1 → S2

Generated by `age-spe-process-discovery`. Received by `age-spe-architecture-designer`.

```json
{
  "mode": "express | architect",
  "target_platforms": ["antigravity", "claude-code"],
  "process": {
    "name": "",
    "description": "",
    "objective": "",
    "trigger": "",
    "steps": [],
    "decisions": [],
    "integrations": [],
    "human_checkpoints": [],
    "input": {},
    "output": {},
    "constraints": []
  },
  "diagram_as_is": "Mermaid code | null"
}
```

**Notes:**

- `diagram_as_is` is mandatory in Architect Mode before closing S1.
- `constraints` includes technical, business, or access limitations.

## 2. Handoff S2 → S3

Generated by `age-spe-architecture-designer`. Received by `age-spe-entity-builder`.

```json
{
  "target_platforms": ["antigravity", "claude-code"],
  "entities": [
    {
      "type": "workflow | agent | skill | command | rule | knowledge-base | script | hook",
      "name": "",
      "description": "",
      "function": "",
      "relationships": [],
      "is_new": true,
      "intricacy_level": "simple | medium | complex"
    }
  ],
  "architecture_diagram": "Mermaid code | null",
  "creation_order": [],
  "reused_skills": []
}
```

**Notes:**

- `creation_order` defines the entity generation sequence in S3.
- `reused_skills` lists existing Skills that will not be regenerated.
- `architecture_diagram` is mandatory in Architect Mode before closing S2.

## 3. Phase metrics object

Maintained by the orchestrator and emitted as part of each handoff (consumers tolerate absence):

```json
{
  "schema_version": "1.1",
  "regenerations": 0,
  "iterations": 0,
  "input_tokens": 0,
  "output_tokens": 0,
  "cache_hit_rate": 0.0,
  "duration_ms": 0
}
```

**Required fields** (always emitted):

- **regenerations:** increment by 1 each time the user chooses option C (regenerate) at a checkpoint.
- **iterations:** increment by 1 each time the user chooses option B (edit/adjust) at a checkpoint.

**Optional observability fields** (emitted when the host supports them):

- **input_tokens / output_tokens:** token usage per agent invocation.
- **cache_hit_rate:** float in [0, 1]. Prompt cache effectiveness for the invocation. Higher is better.
- **duration_ms:** wall-clock duration of the agent invocation.

**Notes:**

- In S3, metrics are the cumulative sum of all entities in the Step.
- Agents emit metrics when they can; consumers tolerate their absence (graceful degradation).

## 4. Context Ledger Schema

Temporary `context-ledger.md` file created and managed by the workflow during execution. Persists the output of each step and allows traceability.

**Location:** `{export-path}/context-ledger.md`.

```markdown
---
system: { system-name }
workflow: { workflow-name }
created: { ISO timestamp }
last_updated: { ISO timestamp }
---

## [Step {N}] — {agent-name} — {completed|in_progress|pending}

### Input received

{filtered context the workflow passed to the agent}

### Generated output

{output produced by the agent — handoff JSON, diagram, files, etc.}

### Metrics

(optional; emitted when the host supports them)

- input_tokens: {N}
- output_tokens: {N}
- cache_hit_rate: {0.0..1.0}
- duration_ms: {N}

### Metadata

- Timestamp: {ISO}
- Step: {N of M}
```

**Writing rules:**

- Always **append** — never overwrite previous blocks.
- One `## [Step N]` block per agent invocation.
- The `status` field is updated in-place (pending → in_progress → completed).
- The **Metrics** block is optional. If the host runtime exposes token counts and timing, agents emit them; otherwise the section is omitted. Consumers must tolerate its absence.

## 5. Context Map Schema

The Context Map is defined within the workflow and declares what context flows between steps:

```markdown
| Destination Step | Consumes from   | Fields / Sections  | Mode     |
| ---------------- | --------------- | ------------------ | -------- |
| {step N}         | Step {M} output | {field1}, {field2} | partial  |
| {step N}         | Step {M} output | \*                 | complete |
```

**Modes:**

- `complete`: the full output of the referenced step is passed as input.
- `partial`: only the listed fields or sections are extracted and passed.

**Notes:**

- A step can consume from multiple previous steps (multiple rows with the same destination).
- The workflow reads the Context Map and builds the filtered input before invoking each agent.
