## Context

This rule ensures that all entity files generated by the Agentic Architect system follow consistent naming conventions. Consistency in names and paths is critical for cross-references between entities to work correctly and for files to be identifiable at a glance.

## Hard Constraints

- Never create an entity file without the correct prefix for its type.
- Never use uppercase letters in entity file or folder names.
- Never use spaces in file names — always use hyphens (`-`).
- Never exceed 64 characters in the `name` field of the frontmatter.
- Never exceed 250 characters in the `description` field of the frontmatter.
- Never use the same name for two different entities, regardless of their type.
- Never reference an entity with a name different from the one defined in its frontmatter.
- If the entity content approaches or exceeds the recommended limit, it MUST NOT remain monolithic. The information must be structured and partitioned by creating support documents in the `/resources` directory and referencing them from the main entity's instructions. A relational and hierarchical system must be prioritized.

## Soft Constraints

- Prefer descriptive and specific names over generic ones (`age-spe-email-classifier` is better than `age-spe-agent-1`).
- The name should reflect the function of the entity, not its position in the flow.
- The frontmatter description should be readable independently and understandable without additional context.
- **Description quality standard ("what + when" pattern):** Every `description` field must answer two questions: (1) **What** does this entity do — specific action, third person; (2) **When** should it be used — trigger context, specific conditions. Include keywords that enable auto-discovery. Example: _"Classifies incoming support emails by department and urgency. Use when an unstructured email arrives and needs to be routed before creating a ticket."_
- When adding, removing, or renaming entities, update `system-overview.md` accordingly to keep the entity inventory current.

| Type             | Prefix           | Example                       |
| ---------------- | ---------------- | ----------------------------- |
| Workflow         | `wor-`           | `wor-customer-onboarding.md`  |
| Agent Specialist | `age-spe-`       | `age-spe-email-classifier.md` |
| Agent Supervisor | `age-sup-`       | `age-sup-output-validator.md` |

## File Structure by Type

```
├── workflows/        → wor-[name].md / age-spe-[name].md / age-sup-[name].md / com-[name].md
├── skills/
│   └── ski-[name]/
│       └── SKILL.md  → the folder name is the identifier
├── rules/            → rul-[name].md
├── knowledge-base/   → kno-[name].md
├── resources/        → res-[name].md
├── scripts/          → scp-[name].sh / scp-[name].py (CC) / scp-[name].md (GA)
└── hooks/            → hok-[name].md
```

## Codex File Types

Codex uses compiled formats for behavioral entities. These are auto-generated by `build-codex.py` and should never be created manually:

| Type             | Codex Format                      | Source                     |
| ---------------- | --------------------------------- | -------------------------- |
| Behavioral       | `.toml` (TOML agent definition)   | `wor-*.md`, `age-*.md`, `com-*.md` |
| Hook fragment    | `.json` (individual config)       | `hok-*.md`                 |
| Merged hooks     | `hooks.json`                      | All `hok-*.json` fragments |
| Configuration    | `config.toml`                     | Generated by build script  |

## Relative Paths Between Entities

When an entity references another, always use relative paths from the root directory:

| Referenced Entity | Relative Path                      |
| ----------------- | ---------------------------------- |
| Skill             | `./skills/ski-[name]/SKILL.md`     |
| Agent             | `./workflows/age-[type]-[name].md` |
| Workflow          | `./workflows/wor-[name].md`        |
| Rule              | `../rules/rul-[name].md`           |
| Knowledge-base    | `../knowledge-base/kno-[name].md`  |
| Command           | `./workflows/com-[name].md`        |
| Resources         | `../resources/res-[name].md`       |
| Script            | `./scripts/scp-[name].sh` or `./scripts/scp-[name].md` |
| Hook              | `./hooks/hok-[name].md`           |
| Repository Index  | `../../repository/[type]-repo.md`  |

## Repository Index File Format

All files with the `-repo.md` suffix inside the `repository/` folder must contain a single Markdown table with the following 4 literal columns:

| Name | Systems where used | Related to | Purpose / Description |
| ---- | ------------------ | ---------- | --------------------- |

**Constraints for this table:**

- **Column 1 (Name)**: Backtick reference to the exact entity name (e.g. `age-spe-email-classifier`).
- **Column 2 (Systems)**: Comma-separated list of all exported {system-name} that consume this entity.
- **Column 3 (Relations)**: Known connected entities, or dash `-` if not applicable.
- **Column 4 (Purpose)**: Concise paragraph describing the operational objective of the piece to promote reuse.
