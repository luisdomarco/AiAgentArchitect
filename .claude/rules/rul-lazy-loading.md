---
name: rul-lazy-loading
description: Establishes the lazy-loading principle for AiAgentArchitect — every entity reads only what it needs at the moment it needs it. The session boot path stays minimal; per-step reads are gated by an explicit condition (active layer, target_platforms, current step). Violations balloon token consumption and slow session startup. Auditors flag any boot-time read above the allowlist or any preventive load of knowledge-base/resources files.
---

## Context

AiAgentArchitect runs as a long-form workflow with multiple agents and many knowledge files (rules, knowledge-base, resources, layer manifests, templates). Loading everything at session start is the obvious-but-wrong default: it inflates the boot context, slows the model's first response, and burns tokens on files that may never be touched. v2 established this principle implicitly by having each agent read only the files relevant to its phase. v3 makes it explicit so future layers do not regress it as the system grows.

## Hard Constraints

- The session **boot path** MUST NOT load anything beyond this allowlist:
  - `system-overview.md` (root index — replaces reading individual entity files)
  - `config/manifest.yaml` (declares which layers are active)
  - `res-architect-execution-phases.md` (orchestrator's operational manual)
  - the most recent `memory/*.md` snapshot for the active project, if any
  - a pre-filled `%Master - Docs/template-input-*.md`, if present
- Layer MANIFESTs (`.agents/layers/{X}/MANIFEST.yaml`) MUST NOT be read at boot. They are read only when `ski-layer-embed` activates the layer, or when the layer is invoked for the first time in the session and its hooks need to be resolved. The boot reads only the `enabled` field of each layer from `config/manifest.yaml` to build the in-memory `active_layers` set.
- Knowledge-base files (`kno-*.md`) MUST NOT be loaded preventively. An agent reads a `kno-*` only when it actually needs the information (e.g. `age-spe-architecture-designer` reads `kno-entity-selection` only when picking entity types in Step 2).
- Resource files (`res-*.md`) follow the same rule: load only inside the step that consumes them. The `res-platform-dispatch` pattern (load templates only for active `target_platforms`) is the canonical example.
- Rule files (`rul-*.md`) are loaded by the agent that the rule applies to, when the agent runs — never preemptively by the orchestrator.
- A new layer that introduces a boot-time read MUST justify it in its MANIFEST description and pass review against this rule.

## Soft Constraints

- Prefer in-memory caches (e.g. the `active_layers` set built once from `manifest.yaml`) over re-reading files mid-session.
- Prefer pre-built indexes (`repository/entities-registry.csv`, autogenerated) over scanning directories at runtime.
- Prefer reading a single index/summary (`system-overview.md`) over reading every file the index points to.
- Document each lazy-load decision in the consumer entity. Example phrasing in an agent file: _"I read `kno-Y` only when I need to pick X."_ Tacit lazy-loading still works, but explicit lazy-loading is reviewable and survives refactors.
- When generating subsystems, replicate this principle inside the subsystem's `wor-*` workflow and agents — don't ship subsystems that load everything at start.

## Rationale

In v2, the average session boot was ~500 lines (~125 tokens × 4 ≈ 500 lines of context). v3's modular design risks ballooning this if every layer's MANIFEST is read at startup (~14 files × ~80–130 lines = ~1,250 extra lines). The `active_layers` cache strategy avoids that, but only if every consumer of layer metadata respects it. Without an enforceable rule, a future contributor adding a 15th layer could naively read its MANIFEST in the orchestrator boot block and silently regress the whole system. This rule makes the principle reviewable and auditable.

## How auditors check this rule

When auditing `wor-agentic-architect.md`, `res-architect-execution-phases.md` or any `age-spe-*.md` agent file:

1. Identify each `read`/`load`/`open` reference in the boot path (Session Start sequence).
2. Verify each reference is in the allowlist above. Flag any other read at boot.
3. Identify per-step reads. Verify each is gated by an explicit condition (e.g., `if active_layers['memory'] then read memory/...`, `if target_platforms includes 'codex' then read res-codex-conventions`).
4. Flag any single read of >200 lines that is unconditional — strong candidate for lazy-loading.
5. For new layers, verify their MANIFEST description does not promise boot-time reads.

## Examples

### Good (allowed)

```
Session Start:
  Read system-overview.md           → covered by allowlist
  Read config/manifest.yaml          → covered by allowlist
  Read res-architect-execution-phases.md → covered by allowlist
  Read memory/2026-04-27-foo.md      → covered by allowlist (latest snapshot)

Step 2 (Architecture Designer):
  if active_layers['context-ledger']:
    Read context-ledger entry for S1   → gated read, OK
  Read kno-entity-selection.md         → just-in-time, OK
```

### Bad (would fail audit)

```
Session Start:
  Read system-overview.md
  Read config/manifest.yaml
  for each layer in manifest.layers_root:
    Read .agents/layers/{layer}/MANIFEST.yaml   ← VIOLATION: preventive load
  Read kno-handoff-schemas.md                    ← VIOLATION: not in allowlist; load at Step 1 instead
  Read all rul-*.md                              ← VIOLATION: rules are loaded by their consumer
```

## Related entities

- `wor-agentic-architect` — must respect the boot allowlist in its Session Start section.
- `res-architect-execution-phases` — defines the canonical Session Start sequence.
- `res-platform-dispatch` — canonical example of conditional load (per `target_platforms`).
- `kno-system-architecture` — describes the index-vs-detail distinction.
- All `age-spe-*` agents — each must scope their reads to what they actually need for their phase.

## Automated audit

Run `python3 scripts/audit-lazy-loading.py` from the repo root to scan all workflows and skills for compliance violations. The auditor reports findings (critical + warnings) without modifying anything; humans decide what to fix. Intended to run in pre-commit hooks or CI when convenient.

```
python3 scripts/audit-lazy-loading.py            # human-readable report
python3 scripts/audit-lazy-loading.py --quiet    # only summary
python3 scripts/audit-lazy-loading.py --json     # machine-readable
```

Exit code: `0` if no critical violations (warnings allowed), `1` if any critical violation present.
