## How it works

Run the main workflow to start a design session:

- **Antigravity:** `wor-agentic-architect`
- **Claude Code:** `/wor-agentic-architect`

For context-aware help at any time: `/help`.

## Active session behavior

**On every interaction, before responding, check this:**

1. Look for the latest snapshot in `memory/` matching the active project (the most recent `*.md` file by mtime).
2. If a snapshot exists with `status: in-progress` and `last_checkpoint` set, **the user is mid-workflow**. Treat any non-slash user message as continuation of that workflow:
   - Read the snapshot to recover `mode`, `last_checkpoint`, and `target_platforms`.
   - Resume the orchestrator's Session Start logic from `res-architect-execution-phases` step 6b (the "I found a previous session" branch).
   - Do **not** require the user to type `/wor-agentic-architect` again to re-enter the flow. The slash command remains available for explicit re-entry, but is no longer mandatory after a pause.
3. If the user message **is** a slash command (`/...`), defer to that command — do not auto-resume.
4. If the user message is unrelated to the workflow (e.g. a meta-question about the system, or a clearly off-topic request), do not force resumption — answer the question directly and ask if they want to resume the workflow.

## Project structure

Three platform implementations of the same agentic system:

- **`.agents/`** — Antigravity (source of truth). Workflows, agents, skills, rules, knowledge-base, resources, scripts, hooks. Plus `.agents/layers/{layer-id}/` for the four bundled layers.
- **`.claude/`** — Claude Code mirror. Bidirectional sync via `bash scripts/sync-dual.sh`. Layer entities are duplicated to canonical paths (`.claude/{commands,agents,skills}/`) for CC discovery, and to `.claude/layers/{layer-id}/` as structural mirror.
- **`.codex/`** — OpenAI Codex compiled output. One-way generation via `python3 scripts/build-codex.py`. Behavioral entities become TOML agents.

## Bundled layers

The Lite edition ships with four small layers, all enabled by default:

- **context-ledger** — append-only trace of each step
- **memory** — cross-session snapshots so you can pause and resume
- **help-router** — `/help` shows context-aware options based on your current phase
- **onboarding** — first-run guided tour

Layers self-describe via `.agents/layers/{layer-id}/MANIFEST.yaml`. Embed/unembed/update is implemented internally by `ski-layer-embed`.

## Active Layers

### context-ledger

- **Context Ledger active**. Append-only trace at `context-ledger/YYYY-MM-DD-HH-MM-{project}.md`.
- Each step records: received input, reasoning trace `<sys-eval>`, output, metadata.
- The orchestrator filters context via the Context Map before injecting into the next agent.
- Handoff JSON schemas defined in `kno-handoff-schemas`.

### memory

- **Memory Layer active**. Persistent snapshots at `memory/YYYY-MM-DD-HH-MM-{project}.md` (~1-2 KB).
- At session start the orchestrator offers to resume the most recent project, switch projects, or start a new one.
- After each approved checkpoint the snapshot is updated automatically (orchestrator-driven).

### help-router

- **Help Router active**. Use `/help` at any time to see context-aware options based on your current phase and project state.
- Behind the scenes, the router reads `repository/entities-registry.csv` and the active Memory snapshot to filter options by active layers.

### onboarding

- **Onboarding active**. On first invocation (no `memory/welcome-shown.md` marker), `wor-onboarding` auto-runs to give a brief tour, verify your install, and offer first-step options.
- Manual invocation: `/wor-onboarding`. Skip the auto-tour: `touch memory/welcome-shown.md`.


## Configuration

Runtime behavior is governed by `config/`:

- **`config.base.toml`** — defaults (versionable, do not edit; override via team/user).
- **`config.team.toml`** — team-wide overrides (versionable). Fields can be locked with `<field>_lock = true`.
- **`config.user.toml`** — personal overrides (`.gitignored`, `0600`).
- **`manifest.yaml`** — installation state (which layers + platforms are active). Generated by `node scripts/install.mjs`. Re-run the wizard to modify.
- **`.resolved.toml`** — merged final config, regenerated by `python3 scripts/resolve-config.py`.

Precedence: `base < team < user < CLI flags`.

## Synchronization model

- **`.agents/` ↔ `.claude/`**: bidirectional via `bash scripts/sync-dual.sh`. The `--prune` flag removes orphaned files in `.claude/` that no longer have a counterpart in `.agents/` (root or any layer). The reverse direction (`.claude/` → `.agents/`) only handles root entities; always edit layer entities in `.agents/layers/{X}/`.
- **`.agents/` → `.codex/`**: one-way via `python3 scripts/build-codex.py`. Behavioral entities (wor-, age-, com-) compile to TOML; everything else copies directly. Never edit `.codex/`; it is overwritten on every build.
- **Context roots** (`CLAUDE.md`, `AGENTS.md`): regenerated by `python3 scripts/build-context-roots.py` (auto-invoked after sync and build).

## Scope boundaries

Strict read boundaries to keep token usage bounded — see [rul-scope-boundaries](.agents/rules/rul-scope-boundaries.md). Summary:

- **Working on AiAgentArchitect itself**: never read inside `exports/` (except `exports/template/`).
- **Working on a specific export**: only read that export's directory; never the parent or other exports.
